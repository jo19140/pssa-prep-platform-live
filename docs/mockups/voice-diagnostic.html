Design notes · voice-first diagnostic

  • No screen requires reading to operate. The buddy speaks every instruction. The text shown is illustrative — a kid who can't read can still complete every task by listening + speaking + tapping pictures.
  • Character with named states. Idle (warm gradient), listening (emerald + pulse ring), speaking (purple + voice waves), confused (amber + thinking face). These states are the visual grammar of the whole voice mode and should be reused identically in voice-practice.
  • Push-to-talk by default, always-listening optional. Hold-to-talk gives kids control over when the program is listening; reduces accidental capture. An accessibility toggle for hands-free mode would unlock it for kids with motor difficulties.
  • "I heard you say…" feedback. Critical for trust. Kid sees what the program transcribed so they can correct if it misheard. Also doubles as a literacy artifact — they see their own speech rendered as text.
  • Picture-based answer choices. When a multiple choice can be answered visually (emotions, scenes, simple concepts), use emojis or images instead of text. Kid who can't read can still answer.
  • "No score is being taken — just try again." Voice mistakes (mishears, mumbles, stage fright) must not affect scores. Diagnostic results should be based on what the kid clearly demonstrated, not on speech-recognition failures.
  • Karaoke-highlight on the read-aloud passage. As the buddy reads, the current word highlights. This is foundational for emergent readers — they see word-sound mapping in real time, building decoding without realizing they're being taught.
  • Existing infra: ReadingCoachPanel.tsx already has MediaRecorder + voice capture. Voice-first mode extends this with TTS (browser SpeechSynthesis API is the cheap MVP; ElevenLabs or similar for a friendlier voice in production) and intent recognition on the kid's responses.
